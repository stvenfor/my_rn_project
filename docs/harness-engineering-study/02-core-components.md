---

## 2.1 上下文工程（新员工手册）

**原则**：恰好获得当前任务所需上下文——不多不少。

| Tier | 加载时机 | 内容示例 | 占用 |
|------|----------|----------|------|
| 1 会话常驻 | 每次自动 | AGENTS.md、项目概览 | 最小 |
| 2 按需 | 子 Agent/技能触发 | 领域知识、专家上下文 | 中等 |
| 3 持久知识库 | Agent 主动查询 | 规格、研究文档、历史会话 | 按需 |

实践要点：

- **地图模式**：AGENTS.md 写导航，不写百科（建议 50–100 行）
- OpenAI：失败即更新 AGENTS.md（动态反馈循环）
- Anthropic：大量 README + 频繁更新的进度文件
- Hashimoto / Ghostty：**每一行对应一次历史失败**

## 2.2 Agent 专业化

**原则**：领域受限工具的专家 > 全权限通用 Agent（运行在 “Smart Zone”）。

| 角色 | 职责 | 工具权限 |
|------|------|----------|
| 研究 | 探索、分析 | 只读 |
| 规划 | 拆任务 | 只读，无写入 |
| 执行 | 实现单个任务 | 限定范围读写 |
| 审查 | 审计、标问题 | 只读 + 标记 |
| 调试 | 修审查问题 | 限定范围修复 |
| 清理 | 对抗熵 | 读写 |

Carlini C 编译器：核心 / 去重 / 性能 / 文档 等专职角色。

---

## 2.3 持久化记忆

**原则**：进度在**文件系统**，不在上下文窗口。新会话从零开始，靠制品重建上下文。

Anthropic 两阶段：

1. **初始化 Agent**：`init.sh` + `claude-progress.txt` + 初始 git  
2. **编码 Agent**：增量进展 + 结构化更新  

编码会话启动清单：

1. `pwd`  
2. 读 `git log` + 进度文件  
3. 读 feature list，选最高优先级未完成项  
4. 启开发服务器，跑基础 E2E  
5. 确认正常后再开新功能  

**关键发现**：feature 状态用 **JSON** 比 Markdown 更稳（不易被不当覆盖）。

---

## 2.4 结构化执行

**原则**：思考与执行分离。序列：**理解 → 规划 → 执行 → 验证**。

- 人工检查点优先审**计划**（比审代码快）
- 规格对 → 实现可靠；规格错 → 在 500 行生成前纠正
- Huntley：规划模式 / 构建模式分离  
- Horthy：Research-Plan-Implement  

---

## 2.5 架构约束（缰绳）

**原则**：写在文档里不够；**不能机械化强制，Agent 就会偏离**。

OpenAI 层级（下层不可反向依赖上层）：

```text
Types → Config → Repo → Service → Runtime → UI
```

Linter 错误信息 = Prompt，三要素：

```text
❌ [什么错了]
✅ FIX: [怎么改，给代码片段]
📖 See: [文档路径]
```

---

## 2.6 反馈循环（智能体审智能体）

- 本地 Agent 自审 → 请求额外审查 → 循环至通过  
- 测试失败带着错误回流模型  
- AI 写的测试若放过带 Bug 代码 → 判定测试无效，强制重思边界  

---

## 2.7 熵管理（垃圾回收）

**原则**：技术债务像高息贷款；持续小额偿还。

- 后台扫描偏差、质量等级、针对性重构 PR  
- **Doc-gardening Agent**：文档与代码不一致 → 自动 PR  
- OpenAI：周五清 AI Slop → 自动化为后台任务（清理吞吐与生成吞吐同比扩展）
