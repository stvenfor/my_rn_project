# 01 · 概念与动机

数据源：[`data/core.json`](./data/core.json)

## 1.1 定义

**Harness Engineering（驾驭工程）**：围绕 AI 智能体设计并构建约束机制、反馈回路、工作流控制与持续改进循环的系统工程实践。

- 不优化模型本身，而优化**模型运行的环境**
- 「Harness」= 马具（缰绳、马鞍、嚼子）——引导强大但不可预测的力量，而非削弱它
- 术语提出：Mitchell Hashimoto，2026-02-05；随后 OpenAI 百万行实验报告正式采用

> anytime you find an agent makes a mistake, you take the time to engineer a solution such that the agent will not make that mistake again in the future.  
> —— Mitchell Hashimoto

潜台词：**失败是环境信号**；正确回应是改环境，不是换更强模型。

## 1.2 三次范式跃迁

| 范式 | 核心问题 | 优化对象 | 交互模式 |
|------|----------|----------|----------|
| 提示词工程 | 怎么把话说清楚 | Prompt 措辞/格式/示例 | 一问一答 |
| 上下文工程 | 怎么给 AI 喂信息 | 文档、代码、历史对话 | 信息注入 → 生成 |
| **驾驭工程** | 怎么让 Agent 可靠工作 | 约束、反馈、控制系统 | **人类掌舵，Agent 执行** |

类比：

| 层 | 比喻 |
|----|------|
| Prompt | 对马喊话 |
| Context | 给马看地图 |
| Harness | 造高速公路：护栏、限速牌、加油站 |

## 1.3 与传统 Agent 框架的关系

```text
┌─────────────────────────────┐
│     Harness（驾驭层）         │  ← 可靠运行：持久化、重放、成本、可观测、恢复
├─────────────────────────────┤
│  SDK / 脚手架 / Agent 框架   │  ← 如何构建智能体（~80% 正被模型吸收）
├─────────────────────────────┤
│           模型               │
└─────────────────────────────┘
```

- 框架解决：**如何构建**智能体  
- 驾驭层解决：**如何可靠运行**  
- 模型逐渐吸收框架约 80% 功能；剩余 20% 正是 Harness 价值所在

## 1.4 为什么需要（证据）

| 信号 | 数值 |
|------|------|
| OpenAI 团队产出 | ~100 万行，**0 行**工程师手写 |
| 团队规模 | 3–7 人，人均每日 ~3.5 PR |
| LangChain（只改 Harness） | Terminal Bench 排名 **30 → 5**，得分 **52.8% → 66.5%** |

结论（多团队独立一致）：**瓶颈在基础设施，不在模型智能。**

## 1.5 Agent 常见失败模式

| ID | 模式 | 表现 | 对策方向 |
|----|------|------|----------|
| F1 | 一步到位（One-shotting） | 单会话塞满功能 → 上下文耗尽 → 半成品 | 功能列表 + 每次只做一个 |
| F2 | 过早宣布胜利 | 有部分进展就宣称项目完成 | JSON feature list 强制选单功能 |
| F3 | 过早标记功能完成 | 写完就标 done，无 E2E | 测试后才标 `passing` |
| F4 | 坏模式放大 | 复制并放大库中坏模式/架构漂移 | 机械化架构约束 + 熵管理 |
